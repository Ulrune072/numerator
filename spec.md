# spec.md — «Числитель» | Спецификация требований

> Версия 1.0 · Основана на CLAUDE.md + draft.md + ответах на уточняющие вопросы.
> Статус: готово к разработке. Все решения по открытым вопросам зафиксированы ниже.

---

## 1. Принятые решения по открытым вопросам

| Вопрос | Решение |
|---|---|
| Повторное прохождение теста | Все попытки сохраняются в `task_attempts`; баллы начисляются только за первую попытку; в личном кабинете отображается лучший результат |
| Пояснение к ответу | Не отображается — только «верно / неверно» без объяснений |
| Формат контента лекций | WYSIWYG-редактор в админ-панели; в БД хранится HTML |
| Доступ к заданиям | Жёсткая блокировка: пользователь должен открыть страницу лекции перед тем, как появится кнопка «Перейти к заданиям» |
| Управление пользователями | Полный CRUD в админ-панели: просмотр, сброс прогресса, смена роли, удаление |
| OAuth | Не реализуется в v1 |
| Интернационализация | Только русский язык |
| Роль «учитель» | Не реализуется в v1 |

---

## 2. Роли пользователей

**Student** — зарегистрированный пользователь с доступом к лекциям, заданиям и личному кабинету.

**Admin** — пользователь с `role = 'admin'`; имеет полный доступ к контенту и управлению пользователями.

## 3. Функциональные требования

### 3.1 Аутентификация

| ID | Требование |
|---|---|
| AUTH-01 | Неавторизованный пользователь, обращающийся к любому маршруту кроме `/login` и `/register`, перенаправляется на `/login`. |
| AUTH-02 | Форма регистрации принимает email и пароль; при успехе создаёт запись в `auth.users` и соответствующую запись в `profiles` с `role = 'student'`. |
| AUTH-03 | Форма входа принимает email и пароль; при неверных данных показывает понятное сообщение об ошибке. |
| AUTH-04 | После успешного входа пользователь перенаправляется на `/dashboard`. |
| AUTH-05 | Восстановление пароля осуществляется через Supabase Auth: пользователь вводит email и получает письмо со ссылкой для сброса. |
| AUTH-06 | Кнопка «Выйти» завершает сессию и перенаправляет на `/login`. |
| AUTH-07 | Пользователь с `role = 'admin'`, обращающийся к `/admin/*`, получает доступ; пользователь с `role = 'student'` перенаправляется на `/dashboard`. |

### 3.2 Лекции

| ID | Требование |
|---|---|
| LEC-01 | Страница `/lectures` отображает карточки всех опубликованных лекций в порядке `order_index`. |
| LEC-02 | Каждая карточка содержит: название, краткое описание и статус («Пройдено» / «Не пройдено»). |
| LEC-03 | Страница `/lectures/[slug]` отображает полный HTML-контент лекции. |
| LEC-04 | Кнопка «Перейти к заданиям» на странице лекции становится активной только после того, как пользователь открыл эту страницу (факт посещения фиксируется в сессии или БД). До этого кнопка недоступна или скрыта. |
| LEC-05 | Неопубликованные лекции (`is_published = false`) не видны пользователям с ролью `student`. |

### 3.3 Задания

| ID | Требование |
|---|---|
| TASK-01 | Страница `/tasks/[lectureSlug]` доступна только если пользователь уже посещал соответствующую страницу лекции; иначе — редирект на `/lectures/[slug]`. |
| TASK-02 | Вопросы отображаются по одному в порядке `order_index`. |
| TASK-03 | Каждый вопрос содержит текст и 4 варианта ответа; только один вариант верный. |
| TASK-04 | После выбора ответа отображается только индикатор «верно ✅» или «неверно ❌» — без пояснений. |
| TASK-05 | Переход к следующему вопросу происходит вручную по нажатию кнопки «Далее», которая появляется после ответа. |
| TASK-06 | После последнего вопроса отображается экран результатов: количество правильных ответов, сумма баллов за текущую попытку, и общий счёт пользователя. |
| TASK-07 | Каждая попытка сохраняется в `task_attempts` (selected_index, is_correct, attempted_at). |
| TASK-08 | Баллы (`score_value` за каждый верный ответ) начисляются в `user_progress.score_earned` только при первом прохождении теста по данной лекции. Повторные попытки в статистику не добавляют. |
| TASK-09 | `user_progress.completed` выставляется в `true` и `total_score` в `profiles` обновляется после первого завершения теста. |
| TASK-10 | Пользователь может перепройти тест неограниченное количество раз; все попытки записываются, но баллы повторно не начисляются. |
| TASK-11 | В личном кабинете для каждой лекции отображается лучший результат (максимум правильных ответов) из всех попыток. |

### 3.4 Личный кабинет (`/dashboard`)

| ID | Требование |
|---|---|
| DASH-01 | Отображается имя пользователя (`profiles.username`). |
| DASH-02 | Отображается текущий статус на основе `total_score` (см. таблицу статусов). |
| DASH-03 | Отображается прогресс-бар до следующего статуса. |
| DASH-04 | Отображается список всех опубликованных лекций с пометкой «Пройдено» / «Не пройдено». |
| DASH-05 | Отображается общий счёт баллов. |

**Таблица статусов:**

| Диапазон | Статус |
|---|---|
| 0–99 | Новичок |
| 100–299 | Ученик |
| 300–599 | Знаток |
| 600–999 | Мастер |
| 1000+ | Числитель ⭐ |

### 3.5 Диагностический тест (`/diagnostic`)

| ID | Требование |
|---|---|
| DIAG-01 | Страница содержит вводный текст и Google Form, встроенную через `<iframe>`. |
| DIAG-02 | Результаты формы не влияют на баллы пользователя и не записываются в БД приложения. |
| DIAG-03 | Страница доступна только авторизованным пользователям. |

### 3.6 Админ-панель (`/admin`)

#### Управление лекциями

| ID | Требование |
|---|---|
| ADM-LEC-01 | Список всех лекций (включая неопубликованные) с колонками: название, статус, `order_index`. |
| ADM-LEC-02 | Создание лекции: поля — название, slug (уникальный), краткое описание, контент (WYSIWYG-редактор), `order_index`, флаг публикации. |
| ADM-LEC-03 | Редактирование любого поля существующей лекции. |
| ADM-LEC-04 | Удаление лекции удаляет также связанные задания и прогресс пользователей (CASCADE в БД). |
| ADM-LEC-05 | Переключение `is_published` без входа в режим редактирования — одним кликом. |

#### Управление заданиями

| ID | Требование |
|---|---|
| ADM-TASK-01 | Просмотр всех вопросов по выбранной лекции. |
| ADM-TASK-02 | Создание вопроса: поля — текст вопроса, 4 варианта ответа, индекс правильного ответа, `score_value`, `order_index`. |
| ADM-TASK-03 | Редактирование любого вопроса. |
| ADM-TASK-04 | Удаление вопроса. |

#### Управление пользователями

| ID | Требование |
|---|---|
| ADM-USR-01 | Список всех пользователей с колонками: username, email, роль, `total_score`, количество пройденных лекций. |
| ADM-USR-02 | Изменение роли пользователя (`student` ↔ `admin`). |
| ADM-USR-03 | Сброс прогресса пользователя: обнуляет `total_score`, удаляет все записи `user_progress` и `task_attempts`. |
| ADM-USR-04 | Удаление учётной записи пользователя вместе со всеми связанными данными. |
| ADM-USR-05 | Перед деструктивными операциями (удаление, сброс) отображается диалог подтверждения. |

## 4. Нефункциональные требования

| ID | Требование |
|---|---|
| NFR-01 | Все маршруты защищены middleware Next.js, проверяющим сессию Supabase при каждом запросе. |
| NFR-02 | `SUPABASE_SERVICE_ROLE_KEY` используется только на сервере и никогда не передаётся клиенту. |
| NFR-03 | RLS включён на всех таблицах согласно политикам, описанным в CLAUDE.md. |
| NFR-04 | TypeScript strict mode включён по всему проекту. |
| NFR-05 | Серверные компоненты Next.js используются по умолчанию; `'use client'` добавляется только при необходимости (хуки, события). |
| NFR-06 | Приложение корректно отображается на мобильных устройствах (responsive). |

## 5. Пользовательские сценарии (use cases)

### UC-01 — Регистрация нового пользователя
**Актор:** Гость  
**Предусловие:** Пользователь не авторизован.  
**Шаги:**
1. Пользователь открывает `/register`.
2. Вводит email и пароль.
3. Нажимает «Создать аккаунт».
4. Система создаёт запись в `auth.users` и `profiles` с `role = 'student'`, `total_score = 0`.
5. Пользователь перенаправляется на `/dashboard`.

**Альтернатива (email уже занят):** Отображается ошибка «Пользователь с таким email уже существует».

---

### UC-02 — Прохождение лекции и теста
**Актор:** Student  
**Предусловие:** Пользователь авторизован, лекция опубликована, ранее тест не сдавался.  
**Шаги:**
1. Пользователь открывает `/lectures` и видит список лекций.
2. Переходит на страницу лекции `/lectures/[slug]`.
3. Читает контент; кнопка «Перейти к заданиям» становится активной.
4. Нажимает кнопку → переходит на `/tasks/[slug]`.
5. Видит первый вопрос и 4 варианта ответа.
6. Выбирает вариант → видит «верно ✅» или «неверно ❌».
7. Нажимает «Далее» → переходит к следующему вопросу.
8. После последнего вопроса видит экран результатов: правильных ответов X из N, заработано Y баллов.
9. Баллы добавляются к `total_score`; `user_progress.completed = true`.

**Альтернатива (попытка открыть `/tasks/[slug]` без посещения лекции):** Редирект на `/lectures/[slug]`.

---

### UC-03 — Повторное прохождение теста
**Актор:** Student  
**Предусловие:** Пользователь уже прошёл тест по данной лекции.  
**Шаги:**
1. Пользователь открывает лекцию, нажимает «Перейти к заданиям».
2. Проходит тест.
3. Новая попытка записывается в `task_attempts`.
4. Баллы не начисляются повторно.
5. В личном кабинете отображается лучший результат из всех попыток.

---

### UC-04 — Просмотр личного кабинета
**Актор:** Student  
**Шаги:**
1. Пользователь открывает `/dashboard`.
2. Видит: своё имя, текущий статус (например, «Ученик»), прогресс-бар до следующего статуса, список лекций с отметками «Пройдено» / «Не пройдено», общий счёт.

---

### UC-05 — Прохождение диагностического теста
**Актор:** Student  
**Шаги:**
1. Пользователь переходит на `/diagnostic`.
2. Читает вводный текст и заполняет Google Form в iframe.
3. Результаты не влияют на баллы и не записываются в БД.

---

### UC-06 — Создание лекции администратором
**Актор:** Admin  
**Предусловие:** Пользователь авторизован с `role = 'admin'`.  
**Шаги:**
1. Открывает `/admin/lectures` → нажимает «Создать лекцию».
2. Заполняет форму: название, slug, краткое описание, контент в WYSIWYG-редакторе, `order_index`.
3. Оставляет `is_published = false`.
4. Сохраняет — лекция появляется в списке, но не видна студентам.
5. После проверки переключает `is_published = true` одним кликом.

---

### UC-07 — Создание вопросов к лекции
**Актор:** Admin  
**Шаги:**
1. Открывает `/admin/tasks` → выбирает лекцию.
2. Нажимает «Добавить вопрос».
3. Вводит текст вопроса, 4 варианта ответа, отмечает правильный, задаёт `score_value` и `order_index`.
4. Сохраняет; вопрос появляется в списке.

---

### UC-08 — Управление пользователем администратором
**Актор:** Admin  
**Сценарий A — смена роли:**
1. Открывает `/admin/users`.
2. Находит пользователя.
3. Меняет роль с `student` на `admin` → сохраняет.

**Сценарий B — сброс прогресса:**
1. Нажимает «Сбросить прогресс» рядом с пользователем.
2. Видит диалог подтверждения.
3. Подтверждает → `total_score = 0`, все `user_progress` и `task_attempts` удалены.

**Сценарий C — удаление пользователя:**
1. Нажимает «Удалить» рядом с пользователем.
2. Видит диалог подтверждения.
3. Подтверждает → учётная запись и все связанные данные удалены.

---

### UC-09 — Восстановление пароля
**Актор:** Гость  
**Шаги:**
1. На странице `/login` нажимает «Забыли пароль?».
2. Вводит email.
3. Supabase Auth отправляет письмо со ссылкой для сброса.
4. Пользователь переходит по ссылке, задаёт новый пароль.
5. Перенаправляется на `/login`.

## 6. Тестируемые требования (acceptance criteria)

| ID | Тест | Ожидаемый результат |
|---|---|---|
| T-01 | Открыть `/dashboard` без авторизации | Редирект на `/login` |
| T-02 | Открыть `/admin` с `role = 'student'` | Редирект на `/dashboard` |
| T-03 | Зарегистрироваться с уже существующим email | Сообщение об ошибке, аккаунт не создан |
| T-04 | Попытаться открыть `/tasks/[slug]`, не посетив лекцию | Редирект на `/lectures/[slug]` |
| T-05 | Пройти тест впервые и проверить `total_score` | Баллы добавлены к профилю |
| T-06 | Пройти тест второй раз и проверить `total_score` | Баллы не изменились |
| T-07 | Пройти тест дважды с разными результатами; проверить отображение в кабинете | Показан лучший результат |
| T-08 | Студент открывает неопубликованную лекцию напрямую по URL | 404 или редирект на `/lectures` |
| T-09 | Набрать 100 баллов; проверить статус в кабинете | Статус изменился с «Новичок» на «Ученик» |
| T-10 | Администратор переключает `is_published = false` | Лекция исчезает из списка у студентов |
| T-11 | Администратор сбрасывает прогресс без подтверждения | Диалог подтверждения обязателен; без него действие не выполняется |
| T-12 | `SUPABASE_SERVICE_ROLE_KEY` в браузерном bundle | Ключ отсутствует в клиентском коде |
| T-13 | Студент обращается к RLS-защищённой таблице чужих данных напрямую | Данные не возвращаются |
| T-14 | Набрать 1000 баллов | Статус «Числитель ⭐» |

## 7. За рамками v1 (не реализуется)

- OAuth (Google и другие провайдеры)
- Тёмная тема
- PWA / Offline-режим
- Таблица лидеров
- Стрики и бейджи
- Форматы заданий помимо multiple choice
- Поиск по лекциям
- Роль «учитель»
- Аналитика ошибок по вопросам
- Email-уведомления о новых лекциях
